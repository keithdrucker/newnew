# Overview

This project is a pnpm workspace monorepo using TypeScript, designed to build a comprehensive IT Service Management (ITSM) solution, branded as **Sidekick**, for a construction-firm client. Modeled on established platforms like Freshservice and Atomicwork, Sidekick aims to streamline IT operations, support, and project management. The product brand is **Sidekick** (logo includes the wordmark) — do NOT use the legacy names "Harmony ITSM" or the placeholder "&lt;Client Name&gt;" anywhere in user-visible UI.

The project encompasses:
- A multi-department ITSM web application (`itsm`) for agents and administrators, providing features like a Dashboard, Ticket Board, People, Agents, Knowledge Base, Assets, and Settings.
- A standalone end-user portal (`support`) for submitting and tracking requests, designed for simplicity and ease of use.
- A robust API server (`api-server`) built with Express and Drizzle, serving as the backend for both applications.
- A project management feature (Microsoft Planner-style boards) integrated within the ITSM application.

The business vision is to provide a tailored, efficient, and scalable ITSM platform to enhance internal service delivery and project coordination for the &lt;Client Name&gt; team, improving operational efficiency and user satisfaction.

- **Initiatives** and **Projects** are proactive improvement work

- Never place Knowledge or Timesheets under Administration.

rendered through a new `NavSubGroup` component that suppresses any
sub-group whose visible items are empty (e.g. for `end_user`).

# System Architecture

## Monorepo Structure
The project utilizes a pnpm workspace monorepo with each package managing its own dependencies. The core packages include `artifacts/itsm` (ITSM web app), `artifacts/support` (end-user portal), and `artifacts/api-server` (backend API).

## Technology Stack
- **Node.js**: 24
- **Package Manager**: pnpm
- **TypeScript**: 5.9
- **API Framework**: Express 5
- **Database**: PostgreSQL with Drizzle ORM
- **Validation**: Zod (v4) and `drizzle-zod`
- **API Codegen**: Orval (from OpenAPI spec)
- **Build Tool**: esbuild (CJS bundle)
- **Frontend**: React, Vite, Wouter, TanStack Query

## Technical Implementations
- **Authentication (Demo)**: The support portal uses `localStorage` for `end_user` ID persistence and `POST /api/session/switch` for session management. No passwords are used in the demo.
- **Role-Based Access Control (RBAC)**:
    - `admin`: Full access to all departments and features.
    - `agent`: Scoped to specific `departmentId` or `visibleDepartmentIds` for board members.
    - `end_user`: Scoped to `reporterId = self`.
    - Board membership includes `owner`, `manager`, `modify`, and `read_only` roles, enforced at the API level.
    - **Per-section RBAC (Apr 2026)** — `board_members.sectionRoles` (jsonb, nullable; defined in `lib/db/src/schema/boardMembers.ts`) lets an admin override an agent's role on a per-area basis within the SAME team. Areas (`BoardSection`): `tickets | operational_tasks | initiatives | projects`. Per-section values (`BoardSectionRole`): the four standard roles plus `"none"` (explicit revoke). When `sectionRoles[section]` is set, **it always wins** — including over the legacy implicit `users.departmentId` "modify" fallback (so a home-dept agent set to `read_only` on Projects is NOT escalated to `modify`). When omitted, behavior falls back to the membership's default `role` column. Resolution lives in `getBoardRole(user, deptId, section?)` (`artifacts/api-server/src/lib/board-access.ts`); section-aware list helpers `sectionVisibleDepartmentIds` and `sectionModifiableDepartmentIds` mirror the same precedence and are used by `tickets.ts` (list + assignee validation + time entries) and `projects.ts` (list visibility + cross-functional write check). Initiatives is the one remaining gap — its routes still use the global `assertAgentOrAdmin` gate (no per-team RBAC at all, pre-existing). The `/settings/boards/:slug` page (`artifacts/itsm/src/pages/board-settings.tsx`) is restructured into 4 stacked area sections; each section's right column renders `<SectionMembersCard>` (`artifacts/itsm/src/components/settings/board-members-card.tsx`) with a 5-option dropdown (Full Control / Manager / Modify / Read only / No access) per agent that PATCHes `sectionRoles[section]` directly. Tickets keeps its existing portal/SLA/categories/business-hours form; the other three areas surface a placeholder Setup card. The "Add agent" dialog seeds the team-default `role` only — per-area overrides are tuned afterward via the dropdowns. "Remove from team" deletes the membership row entirely (membership is shared across all 4 areas).
- **Ticket Management**:
    - Ticket key format: `INC-###` (incident) / `REQ-###` (request).
    - Per-department settings for portal, SLA, business hours, and auto-assignment.
    - **Default Ticket Board (per-user)**: `usersTable.defaultTicketBoard` (text, nullable) holds a department slug; `null` means All Tickets. The Tickets page (`artifacts/itsm/src/pages/tickets.tsx`) auto-redirects from `/tickets` to `/tickets/dept/<slug>` on mount when this is set, and exposes a "View Settings" dropdown (`button-view-settings` / `select-default-board`) plus an inline board switcher (`select-board`) so users can change boards manually. The preference is exposed on `Session.defaultTicketBoard` and updated via `PATCH /api/me/preferences` (`updateMePreferences`).
    - **Operational Tasks (Apr 2026)** — IT-initiated, time-based work. Single object table `operational_tasks` (`lib/db/src/schema/operationalTasks.ts`). `type` is `recurring | one_time`; `frequency` is required for recurring (`daily | weekly | bi_weekly | monthly | quarterly | bi_annual | annual | multi_year`) and must be null for one-time. `nextDueDate` is a calendar date (no TZ). Status enum is intentionally minimal: `scheduled | in_progress | completed`. **`overdue` is NEVER stored** — it's derived at read time as `nextDueDate < today AND status != "completed"` and surfaced on every API response as `isOverdue: boolean`. `POST /api/operational-tasks/:id/complete` is the ONLY way to reach `completed`: it sets `completedAt`/`completedById` atomically and, for recurring tasks, inserts a fresh `scheduled` row with `nextDueDate` advanced by the frequency interval, the same checklist (with `done` flags reset), the same owner/name/description, and the chain's shared `seriesId` (= the original instance's id). PATCH refuses any update on a completed task (409) and only accepts `scheduled | in_progress` for `status` (the spec's UpdateInput enum enforces this — admins cannot manually set `completed` or `overdue`). Default list sort is "overdue first then `nextDueDate` asc" via a `CASE WHEN ... THEN 0 ELSE 1 END` ORDER BY. RBAC uses `getBoardRole(user, deptId, "operational_tasks")` + the same `sectionVisibleDepartmentIds` / `sectionModifiableDepartmentIds` helpers as Tickets/Projects (so per-section overrides on `board_members.sectionRoles` work for this section too); end_users have no access. Checklist items live inside the row's `checklist` jsonb (each: `id`, `text`, `done`, optional `assigneeId` + `dueDate`); the route back-fills missing UUIDs on insert. UI: `artifacts/itsm/src/pages/operational-tasks.tsx` is a Tickets-style filter bar (search / status / owner / frequency / type / due-window: today|week|overdue) over a flat table (Task Name | Frequency | Type | Next Due | Owner | Status), row-click opens an inline detail dialog with editable metadata + description + checklist editor + a "Mark complete" button (one-time tasks get a confirmation step since completion is final).
    - **Per-page default-team selectors + saved views (Improvements + Operational Tasks)** — Apr 2026: the same default-team pattern is mirrored for Initiatives, Projects, and Operational Tasks via three new nullable `usersTable` columns: `defaultInitiativeBoard`, `defaultProjectBoard`, `defaultOperationalTaskBoard`. All four are exposed on `/api/session` and writable through `PATCH /api/me/preferences`. Each page auto-redirects from its bare URL to `/<section>/dept/<slug>` when the corresponding default is set; `?all=1` is the explicit-all override that suppresses the redirect. Initiatives, Projects, **and Risk Register** also support **saved views** via a generic `boardViewsTable` (`lib/db/src/schema/boardViews.ts`) partitioned by `scope` (`"initiative" | "project" | "operational_task" | "risk"`), keyed unique on `(userId, scope, name)`, with a free-form `config` jsonb (per-page filters/sort/team scope). CRUD lives at `/api/board-views?scope=…` (`artifacts/api-server/src/routes/boardViews.ts`); the page header in each section renders `Section · Team ▾ › View ↕` dropdowns so users can save / apply / promote-to-default / delete views and see the active default auto-apply on first load. Tickets keeps its original dedicated `ticket_views` table — the new generic table is only for the three other sections. **Quirk:** orval generates a strict Zod schema for `BoardViewConfig` that strips unknown keys; the route writes raw `req.body.config` for POST/PATCH and skips `ListBoardViewsResponse.parse` on LIST so arbitrary per-page filter keys (e.g. `priority`, `category`, `riskLevel`) round-trip intact.
    - **Create Ticket dialog** (`artifacts/itsm/src/components/create-ticket-dialog.tsx`): launched from the primary "Create ticket" button (`button-create-ticket`) in the Tickets page header; submits via `useCreateTicket` and invalidates `/tickets` queries on success. Defaults `departmentId` from the current board's `deptSlug`, falling back to the user's department. **Server-side `POST /api/tickets` ignores `req.body.reporterId` and always binds the reporter to the authenticated user** (`getCurrentUser(req).id`) — the field is accepted only for spec compatibility, never trusted, to prevent ticket-impersonation.
- **Project Management (Phase-driven lifecycle)** — Apr 2026 rework:
    - Single fixed 7-phase board at `/projects` (no per-department boards): `backlog_needs_assignment` → `planning` → `in_progress` → `completed` → `closed`, with `on_hold` and `cancelled` as side states. `previousActivePhase` is captured on hold so Resume returns to the right lane. The legacy department-bucket board (`departmentBucketsTable`, `projectsTable.bucketId`) is preserved but no longer driven by the UI.
    - **Phase-content model** (per spec): Backlog owns assignment + timeline (`ownerId`, `assignedTeam`, `priority`, `startDate`, `endDate` = "Anticipated completion date"). Planning is checklist-only (`planningNotes` + `checklist[]`); start/end carry forward as read-only. In Progress shows the original dates read-only — no separate re-estimate field; the actual finish is captured automatically as `completedAt` when Mark Completed is clicked. **Closeout is two-step**: `completed` is the **paperwork phase** (no required body; auto-captures `completedById` + `completedAt`; the dialog renders editable `completionSummary` + `keyTakeaway` Textareas with a "Save draft" button). `closed` is the **locked archived phase** (its own board lane); transitioning into it requires non-empty `completionSummary` AND `keyTakeaway` (either already on the row from PATCH edits, OR in the request body) and auto-captures `closedById` + `closedAt`. Either `completed` or `closed` can be reopened back to `in_progress`.
    - **Phase-transition gates** (POST `/api/projects/:id/phase`): legal transitions are enumerated in `PHASE_TRANSITIONS`: `completed: ["closed", "in_progress"]`, `closed: ["in_progress"]`. `backlog → planning` requires both dates with `endDate > startDate` (else 400 "Anticipated completion date must be after start date."). `* → completed` is **bare** (no required body — closeout text is filled in afterwards via PATCH on the Completed view). `* → closed` requires non-empty summary AND key takeaway. Resume from `on_hold` must return to `previousActivePhase`. Every transition writes a `project_audit_events` row in the same `db.transaction(tx)` as the column update via `writeAudit(tx, ...)`; `writeAudit` accepts a tx executor so phase changes, all four checklist endpoints, and project-create insert are all atomic.

Each `ChecklistItem` is `{ id, position, text, done, assigneeId|null, dueDate|null }` (the latter a YYYY-MM-DD string) with stable UUIDs back-filled lazily by the API on first interaction; the editable Planning/In-Progress checklist UI exposes a per-item assignee picker and date input alongside the text.

The phase modal has phase-specific required fields (`holdReason` for on_hold; `completionSummary` + `keyTakeaway` for **closed** (not completed); `cancellationReason` for cancelled). `* → completed` is bare in the modal too.

Treatment tab uses `<RiskWorkflowApproval>` (`artifacts/itsm/src/components/risk-workflow-approval.tsx`) — a parameterized clone of the initiative approval component bound to risk runs via `useStartRiskWorkflowRun` + `module: "risks"`.
    - **Phase-aware detail dialog (May 2026)**: Opening a risk lands on the tab that matches its lifecycle phase (`identified→Overview`, `under_analysis→Analysis`, `under_treatment→Treatment`; everything past treatment falls back to Overview). The three phase tabs (Overview / Analysis / Treatment) are color-coded — completed phases render **green**, the current phase renders **amber/yellow**, and future phases stay default. `Linked Work` and `History` are not part of the lifecycle and never get colored. **All three phase tabs are editable in any status**, so users can refine a previous phase's content (e.g. tweak the title or analysis notes after the risk has moved to Treatment). Each tab has its own Save button; the API permits these field updates at any status — only status TRANSITIONS are gated by `TRANSITIONS`. The form state for **all three** editable tabs is **lifted to `RiskDetailContent`** with `forceMount` `TabsContent` (Radix's default unmounts inactive tab content) + `data-[state=inactive]:hidden` so unsaved edits survive tab switches. `RiskDetailContent` computes per-tab dirty flags + a combined `isAnyDirty` and exposes a single `saveAllDirty(): Promise<boolean>` to the parent `RiskDetailDialog` via a ref-based `registerSaveHandler`. A sticky bottom-right footer shows "You have unsaved changes" whenever any phase tab is dirty; closing the dialog while dirty opens an `AlertDialog` with **Keep editing / Discard / Save changes** (Save runs `saveAllDirty()`).
    - **Analysis tab structure (May 2026)** — Restructured from free-text Impact Scope + Business Impact into 5 explicit sections (each rendered via the local `AnalysisSection` wrapper):
        1. **Risk Scoring** — `likelihood` + `impact` (low/medium/high/critical) + computed-rating badge.
        2. **Impact Assessment** — `employeeImpact` (none/low/medium/high), `financialImpact` (free text — supports ranges like "$50K–$100K"), `operationalImpact` (low/medium/high), `complianceImpact` (yes/no).
        3. **Asset Context** — `assetType` (physical/digital/process/vendor), `assetValue` (free text, $ amount or range), `assetCriticality` (low/medium/high/very_high), plus `exposureFactor` (EF; accepts "25%" or "0.25") and `annualRateOfOccurrence` (ARO; events per year). When all three of `assetValue`, `exposureFactor`, and `annualRateOfOccurrence` parse as numbers, the UI shows the computed **SLE** (= Asset Value × EF) and **ALE** (= SLE × ARO) below the inputs. Inputs are stored as TEXT NOT NULL DEFAULT '' so estimates and ranges are preserved verbatim.
        4. **Risk Factors** — `threats` + `vulnerabilities` (newline-delimited bullets in two textareas).
        5. **Analysis Notes** — single notes textarea.
       The legacy `impactScope` + `businessImpact` columns still exist in `risksTable` (for back-compat / historic data) but are no longer surfaced anywhere in the UI. **`canMoveToTreatment` is now relaxed to require only `likelihood + impact`** — the additional structured fields are advisory, not gating, since the heavy gate is the Team Manager approval at the Treatment phase. All new fields are stored as `text NOT NULL DEFAULT ''` to avoid back-fill pain.
    - **Treatment tab refinements (May 2026)** — The Treatment tab now opens with an amber **"This decision requires Team Manager approval"** banner (non-dismissible info, `data-testid="banner-treatment-approval"`). When `treatmentDecision === "mitigation"`, **five** additional required fields render: `mitigationSummary` (textarea), `mitigationProsCons` (textarea), `mitigationEstimatedCost` (free-text input — supports ranges), **`mitigationControlType`** (select: `security_control` | `compensating_control`), and **`mitigationControlDescription`** (textarea — describes the specific Security Control or Compensating Control to implement). Saving as draft is always allowed; the **approval-opening** call (`POST /api/risks/:id/workflow-runs`) is server-gated in `routes/workflows.ts` to require all five mitigation fields non-empty when the decision is mitigation (returns 400 otherwise), and the same precondition is re-checked at decision finalization (when an approver approves a mitigation run) to guard against stale runs whose fields were cleared after the run was opened. A small amber hint above `<RiskWorkflowApproval>` warns the user when those fields are still missing. Acceptance / Transfer / Avoidance branches keep their existing single-field forms.
        - **Control carries into the Project**: When a mitigation run is approved and the auto-created Project is built (`workflows.ts` finalization), the project description is composed as: original risk description → "Mitigation approved for risk: …" → "**Security Control** / **Compensating Control** to implement: <controlDescription>" → optional decision notes. The control type + description are also written into the `created_from_risk` project audit event's `detail` JSON for traceability. This is the mechanism by which a Security/Compensating Control "goes into the project" the project team will execute.
    - **Linked Work empty state (May 2026)** — Updated to the spec phrasing: *"No linked work yet. Projects are only created for approved Risk Mitigation."* Reinforces that other treatment decisions (acceptance / transfer / avoidance) intentionally do **not** spawn a Project.
- **API Server Conventions**:
    - Query strings are processed by `src/lib/queryCoerce.ts` for numeric parameters.
    - Drizzle's `inArray()` is used for list filters.
    - Atomic increment for Knowledge Base article views.
    - Full-text search uses the `q` query parameter.

## Feature Specifications
- **Sidekick / ITSM (`itsm`)**:
    - **Dashboard**: KPIs, status counts, charts, top agents, recent breaches. Filterable.
    - **Ticket Board**: Collapsible groups for "All Tickets" and 13 departments, showing open ticket counts.
    - **Tickets table**: Columns are `ID, Priority, Risk Level, Status, Title, User, Level, Category, Created, Last Update, SLA`. Risk Level uses color-coded badges (gray = low, yellow = medium, orange = high, red = critical).
    - **Tickets filters**: Status, Priority, Risk Level, Support Level, Category, SLA, Root Cause, Resolution, Created Date, Last Update Date, Assignee — all persisted into named ticket views via `TicketViewConfig`.
    - **Risk Rules**: Admin page at `/settings/risk-rules` (also via Settings → Automation card) maps a ticket category → default risk level via the `risk_rules` table. New tickets created with a matching category default to the configured risk level (overridden by an explicit selection in the create dialog).
    - **Workflows**: Admin page at `/settings/workflows` (Settings → Automation card → Workflows tile) — generic, module-aware rule engine stored in `workflows` / `workflow_runs` / `workflow_run_approvers` / `workflow_audit_events`. Each workflow has WHEN (trigger), IF (conditions JSON), THEN (actions JSON), Approvals (approver kind + targets, approval type single/all/any, optional decision rationale), Notifications, and Status (Draft/Active/Inactive). Modules supported today: Tickets and Initiatives (extensible to Projects/Changes/Risks via `lib/workflow-options.ts` option lists). Phase-1 execution wires Initiative approvals: from an Under Review initiative, an admin clicks "Start Approval Workflow", picks an active Initiatives/Approval workflow, which creates a `workflow_run` with one approver row per resolved user. Approvers record approve/reject/defer + rationale; resolution rules (single = first decision wins; all = unanimous; any = first approve) cascade back through the same allowed-transition path on the parent initiative (approve → `approved` + auto-create Project; reject → `rejected_deferred` w/ `finalDecision=reject`; defer → `rejected_deferred` w/ `finalDecision=defer`). All actions write `workflow_audit_events` (`created/updated/activated/deactivated/triggered/approver_decided/resolved`). Initiative detail dialog shows the active run with approver rows and a past-runs list.
    - **Ticket detail**: Inline editable Root Cause and Resolution panels (save on blur, agent/admin only); Risk Level + Category editable in the sidebar; SLA shown as on-track / breached.
    - **Admin Sections**: People, Agents, Knowledge Base, Assets, Settings, Applications (software catalog), Vendors.
    - **Settings IA (May 2026)** — Settings is restructured into a 6-category responsibility-based tree, driven by a single source of truth at `artifacts/itsm/src/components/settings/settings-catalog.ts`. Categories: System & Defaults, People & Access, Service Configuration, Customer Experience, Technology & Data, Security & Governance (admin-only). Shell is `settings-layout.tsx` (left rail with role-aware gating) + `settings-stub.tsx` (Coming-soon body for unbuilt leaves). Routes live in `App.tsx`: `/settings` redirects to `/settings/system-defaults`; legacy `/settings/agents`, `/settings/risk-rules`, `/settings/workflows*`, `/settings/boards/:slug` all redirect into the new IA. Wired leaves today: Appearance (System & Defaults), Agents (People & Access), Teams (People & Access), Global Workflows + Automation Rules hub + Risk Rules (Service Configuration). Everything else renders SettingsStub. Generic dispatchers are `pages/settings/category-page.tsx` (renders the category landing card grid) and `pages/settings/leaf-page.tsx` (matches `(category, page)` and inlines wired bodies, falling back to SettingsStub).
    - **Per-team Work Types (May 2026)** — Each team opts into a subset of 5 canonical work types (`tickets | operational_tasks | initiatives | projects | timesheets`; **not** Risks) via the per-team page at `/settings/people-access/teams/:slug` (Team Overview / Members / Work Types tabs, in `pages/settings/team-page.tsx`). Persistence is the `team_work_types` table (`lib/db/src/schema/teamWorkTypes.ts`) with `(departmentId, workType)` unique, `isEnabled` (default true), `requiresTimeTracking` (default false). API: `GET /api/departments/:id/work-types` (lazy-creates the 5 default rows on first read) and admin-only `PATCH /api/departments/:id/work-types/:workType` enforcing the rule that `requiresTimeTracking=true` requires `isEnabled=true` (400 otherwise — checked against effective post-merge enablement, not just the request body) and that disabling a work type forces `requiresTimeTracking=false`. Each Work Type card shows Enable + Require Time Tracking switches (time tracking switch is disabled and visually muted while the work type is off) plus a "Manage {WorkType}" button visible only when enabled. Manage page is `pages/settings/team-work-type-page.tsx` with a left section sub-nav whose contents come from `WORK_TYPE_SECTIONS` in `components/settings/work-types-catalog.ts` (per-spec section lists: Tickets has SLAs + Visibility; Initiatives/Projects have Approval Rules with "Uses Team Manager" hint; Projects also has Milestones; Timesheets explicitly notes "no submission/no approval"). Tickets reuses the existing fully-wired `TicketsSetupForm` (exported from `pages/board-settings.tsx`) for its SLAs / Notifications / Automation / Categories / Requirements sections; everything else is a section-level stub today.
